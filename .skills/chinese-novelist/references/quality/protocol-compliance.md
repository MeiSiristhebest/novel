## 路径约定 (Hierarchical Layout & Sliding Window)

本技能库 **仅支持新版工程布局** 并且全面兼容多卷分层架构。约定如下：

- 全局设定根视图：`novels/[书名]/00-全局设定/`
  - `00-策划书.md` / `01-世界观圣经.md` / `02-人设档案.md` / `04-伏笔追踪表.md` / `05-时间线.md` / `06-细节追踪表.md` / `07-项目状态.md`
- 当前卷滑动窗口视图：`novels/[书名]/[当前卷目录]/[大纲文件].md`
- 章节总纲索引：`novels/[书名]/03-分章细纲.md`（由 `.context/scene_list.json` 同步衍生）
- 运行时单一真相源（SSOT）：`novels/[书名]/.context/`（尤其 `STATE.md`、`scene_list.json`、`VOLUME_XX.md`、`RUNNER_STATE.json`）

如果你的项目仍是根目录平铺布局，请先完成迁移（建议使用 `tools/sync_project_views.py` 与 `tools/migrate_novel_state.py`），再开始写作。

## Phase 1：上下文加载 (Context Loading - Progressive Disclosure)

**🧠 AI 渐进式读取指令 (Critical AI Action)**：
为了防止上下文过载或 AI 幻觉，你**必须 (MUST)** 采用「层层展开、渐进式披露」的手段读取 `novels/[书名]` 目录，严禁一次性把所有文件全部吐入 Context。具体执行步骤如下：

### 1.1 第一层：宏观世界与定位 (Global Base)
- **探查**：首先读取 `novels/[书名]/00-全局设定/00-策划书.md` 和 `01-世界观圣经.md` -> 确立核心基调、卖点与力量规则。
- **定位**：读取 `novels/[书名]/00-全局设定/07-项目状态.md` 以及 `.context/STATE.md` -> 获取当前正在写作的【卷名】和【章节编号】。

### 1.2 第二层：卷级视野截取 (Volume Scoping)
- **探查卷面**：使用工具查阅 `novels/[书名]/[当前卷目录]/00-卷规划.md` 或 `.context/VOLUME_XX.md` 获取本卷的核心任务与情感基调。
- **索引章节**：**不要全量读取大细纲！** 利用工具（如 `grep_search` 或 `view_file`）在 `novels/[书名]/03-分章细纲.md` 或 `.context/scene_list.json` 中仅抓取**当前章、前一章、后两章**的上下文，完成剧情滑动窗口。

### 1.3 第三层：弧光片段与角色精准投喂 (Arc & Character Injection)
- **加载 Arc 细纲**：根据章节编号，读取当前卷下具体的 Arc 细纲（如 `01-卷1-末日副本/03-第一卷-107章细纲-Arc1.md`），重点读取本章的【信息投放】与【伏笔】要求，确保底层作者指引百分百生效。
- **角色定向加载**：先读取当前卷的【角色名单】（如 `01-卷1-末日副本/02-角色-卷1-末日副本.md`）了解人物阵营。然后，**仅针对本章出现的核心人物**，再去 `novels/[书名]/00-全局设定/角色库/`（如 `陆存真.md`、`顾时雨.md`）单独读取他们的角色档案。切勿一次性加载所有角色！

### 1.4 第四层：状态与动态对齐 (Tracking Matrix)
- **读取记录**：通过工具浏览 `novels/[书名]/00-全局设定/04-伏笔追踪表.md`、`05-时间线.md` 和 `06-细节追踪表.md`，重点核对与本章情节相关的前置物品、伤势和时间戳，确保因果逻辑连续。
- **平台验证**（选做）：读取 `.context/PLATFORM_STRATEGY_CARD.md` 确认本章是否触碰合规红线或爽点要求。

### 3.2 写作心流构建 (Flow State Pre-computation)
**🧠 AI 强制指令**：在输出任何正文之前，必须在 `<thinking>` 标签中输出以下思考（建立心流）：
1. 确定本段落/章节的情绪水位线（平静/紧张/爆发/余韵）。
2. POV主角当前的底层欲望与核心压力是什么？
3. 如何利用不合理（Anti-Logic）或留白制造反差感？
4. 确认接下来的行文节奏（该Tell还是该Show）。

### 3.3 动笔执行
- [ ] 根据以上心流状态，自然地流淌出正文，忘掉机械的指标，相信直觉。

## Phase 4：质量控制 (Quality Control)

### L1：核心规范（强制 MUST）
- [ ] **AI词汇扫描**：检查烂俗转折词等。
- [ ] **情绪直给检查**：检查是否用了“展示而非讲述”。
- [ ] **对话个性化检查**：角色发音是否符合档案。
- [ ] **动态专项质检**：基于项目约束条件（如群像剧的高光密度检查，通过提取 `00-全局设定/00-策划书.md` 动态应用）。

### L2：80分全维评估（建议 SHOULD）
**评分表**（每项1-10分）：

| 维度 | 评分 | 说明 |
|:-----|:-----|:-----|
| 开头吸引力 | /10 | 前3段是否抓住读者？ |
| 情节推进 | /10 | 本章是否推进主线？ |
| 人物塑造 | /10 | 人物行为一致且有深度？ |
| 对话质量 | /10 | 对话自然且推动情节？ |
| 悬念设置 | /10 | 结尾钩子够硬？ |
| 节奏控制 | /10 | 张弛有度？ |
| 展示而非讲述 | /10 | 用行动/对话而非陈述？ |
| 语言质量 | /10 | 无AI痕迹，用词精确？ |
| **总分** | **/80** | **>60可交付，>70优秀** |

> 参考 `references/quality/genre-quality-checklist.md` 执行流派专项检查

## Phase 5：状态更新与一致性维护 (State Update)

> 参考 `references/guides/consistency-engine.md` 的 **增量更新机制**

**🚨 每章写完后，必须 (MUST) 更新以下文件：**

### 5.1 必更新文件 (双向记录与涌现捕获)

**🧠 AI 记录指令 (Critical AI Action)**：
写作**绝不是**单纯执行细纲的机械劳动。在成文过程中涌现的“神来之笔”（如未计划的新配角、因情绪拉满而说出的新设定词汇、为了画面感临时加的道具等），**必须 (MUST) 被反向记录到追踪表中**！

| 文件 | 动态更新内容要求 |
|:----|:---------|
| `00-全局设定/07-项目状态.md` | 当前章节+1、场景类型、总字数累计 |
| `.context/VOLUME_XX.md` | 章节状态：⏳ → ✅。**用写好的实际表现更新剧情摘要**，而不仅仅是复制大纲。 |
| `03-分章细纲.md` 或 `scene_list.json` | 标记本章完成。如果实际写作中出现了“破局动作变更”或“情绪高光偏离”，**必须用实际高光覆盖原计划，并标注【剧情进化】**。 |
| `00-全局设定/04-伏笔追踪表.md` |不仅要标记已回收的伏笔。更要**主动摘录你在本章描写中临时起意埋下的暗示或诡异细节**，登记为【新埋伏笔】，确保未来必有回响。 |
| `00-全局设定/05-时间线.md` | 记录时间推进，如果有未计划的倒叙记忆被写出，也要登入。 |
| `00-全局设定/06-细节追踪表.md` | 更新人物/物品/地点的状态变化。**若创造了新名词、新招式、新道具，必须在此建档。** |

> SSOT 反向同步规则：AI 在正文中的任何自由发挥，只要不违背原则，一旦写出，就必须反向写回上述 `.context` 和 全局设定 中，使后续 AI 能够接续你的“灵光一闪”。

### 5.2 一致性校验

更新前执行快速校验：
- [ ] 本章出场人物是否都还活着/在场？
- [ ] 人物称呼是否与之前一致？
- [ ] 物品使用是否之前已获得？
- [ ] 消耗品数量是否正确？
- [ ] 时间流逝是否合理？
