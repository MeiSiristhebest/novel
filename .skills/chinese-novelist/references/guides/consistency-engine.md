# AI长文一致性引擎 (Consistency Engine for AI)

> **核心问题**：AI写长篇小说时，最容易犯的错误是**遗忘**和**矛盾**。
> 写到第50章时，AI可能忘了第3章主角断了右手，结果又写"他用右手握剑"。
> 本引擎提供系统化的机制来防止这类问题。

---

## 一、强制上下文加载机制 (Context Loading Protocol)

### 每次写新章节前，AI必须执行以下步骤：

```
[写前加载清单]

1. □ 读取 `00-全局设定/06-细节追踪表.md` —— 检查所有人物/物品/地点的当前状态
2. □ 读取 `00-全局设定/05-时间线.md` —— 确认当前故事时间点
3. □ 读取 `00-全局设定/04-伏笔追踪表.md` —— 检查未解决的伏笔
4. □ 读取上一章最后500字 —— 确保衔接自然
5. □ 读取 `03-分章细纲.md` 中本章的计划 —— 确认本章目标
```

### 前情提要模板 (Previously On...)

每次写新章节时，AI应在内部生成以下摘要（不写入正文）：

```markdown
## 第[X]章 写前摘要

### 上一章发生了什么
- [一句话概括上一章核心事件]
- [上一章结尾的悬念/钩子]

### 当前人物状态
| 人物 | 位置 | 身体状态 | 情绪状态 | 持有物品 |
|-----|------|---------|---------|---------|
| 主角 | [?]  | [?]     | [?]     | [?]     |

### 本章必须处理的事项
- [ ] 回应上一章的悬念：[?]
- [ ] 推进伏笔：[?]
- [ ] 本章核心目标：[?]

### 禁止事项（基于细节追踪表）
- ❌ [例如：主角右手已断，不能用右手做任何事]
- ❌ [例如：小红已死亡，不能再出场]
```

## 二、常见矛盾类型与预防 (Common Contradiction Types)

### 1. 身体状态矛盾
| 错误类型 | 示例 | 预防措施 |
|---------|------|---------|
| 伤残遗忘 | 断了的手又出现 | 在细节追踪表标记"永久状态" |
| 体力透支 | 刚说精疲力竭，下段生龙活虎 | 记录疲劳度，需要恢复事件 |
| 外貌变化 | 第3章黑发，第30章金发 | 锁定外貌描述 |

### 2. 物品状态矛盾
| 错误类型 | 示例 | 预防措施 |
|---------|------|---------|
| 凭空出现 | 从未提过的神器突然拿出 | 物品需要"获取事件" |
| 消失遗忘 | 丢掉的剑又出现 | 记录物品转移 |
| 数量错误 | 说只有3颗丹药，用了5颗 | 追踪消耗品数量 |

### 3. 时间线矛盾
| 错误类型 | 示例 | 预防措施 |
|---------|------|---------|
| 时间倒流 | 昨天发生的事变成三天前 | 维护时间线文档 |
| 年龄错误 | 10年前主角15岁，现在20岁 | 锁定出生年份 |
| 季节混乱 | 冬天突然变成夏天 | 记录季节变化 |

### 4. 人物关系矛盾
| 错误类型 | 示例 | 预防措施 |
|---------|------|---------|
| 称呼变化 | 一会儿叫"师父"一会儿叫"老师" | 锁定称呼规则 |
| 关系遗忘 | 忘了A和B是亲兄弟 | 维护关系图谱 |
| 死而复生 | 死掉的配角又出现 | 标记"已死亡" |

### 5. 设定矛盾
| 错误类型 | 示例 | 预防措施 |
|---------|------|---------|
| 能力膨胀 | 设定只能飞100米，飞了1000米 | 锁定能力上限 |
| 规则破坏 | 设定魔法需要咒语，突然心念一动就发动 | 参照世界观圣经 |
| 货币混乱 | 一会儿1金币很多，一会儿不值钱 | 锁定经济锚点 |

## 三、写后一致性检查清单 (Post-Writing Consistency Check)

每章写完后，执行以下检查：

```markdown
## 第[X]章 一致性检查

### 人物检查
- [ ] 本章出场人物是否都还活着/在场？
- [ ] 人物称呼是否与之前一致？
- [ ] 人物外貌描写是否与之前一致？
- [ ] 人物能力表现是否在设定范围内？

### 物品检查
- [ ] 本章使用的物品是否之前已获得？
- [ ] 消耗品数量是否正确？
- [ ] 重要物品的位置是否合理？

### 时空检查
- [ ] 时间流逝是否合理？
- [ ] 地点转换是否有交代？
- [ ] 季节/天气是否连贯？

### 伏笔检查
- [ ] 是否回应了之前埋下的伏笔？
- [ ] 新埋的伏笔是否记录？

### 设定检查
- [ ] 是否违反了世界观圣经中的规则？
- [ ] 力量体系是否自洽？
```

## 四、增量更新机制 (Incremental Update Protocol)

每章写完后，必须更新以下文档：

| 文档 | 更新内容 |
|-----|---------|
| `00-全局设定/05-时间线.md` | 添加本章的时间节点和事件 |
| `00-全局设定/06-细节追踪表.md` | 更新人物/物品/地点状态变化 |
| `00-全局设定/04-伏笔追踪表.md` | 标记已揭示的伏笔，添加新伏笔 |
| `03-分章细纲.md` | 标记本章完成，填写实际摘要 |

---

## 五、紧急矛盾修复流程 (Contradiction Recovery)

当发现矛盾时：

1. **评估严重程度**
   - 轻微（读者可能不会注意）→ 下一章自然修正
   - 严重（明显穿帮）→ 必须回溯修改

2. **修复策略**
   - **软修复**：在后文通过新情节解释（如：原来是幻觉/替身）
   - **硬修复**：回到前文修改矛盾处

3. **记录教训**
   - 将此类矛盾添加到"禁止事项"列表
   - 更新细节追踪表防止再犯
